# 1. Purpose

Настоящий документ определяет фундаментальные принципы разработки Voxarium.

Все архитектурные, проектные и технические решения MUST соответствовать настоящему документу.

Любой документ спецификации, противоречащий настоящему документу, считается ошибочным.

---

# 2. Project Vision

Voxarium является профессиональной настольной системой полного цикла производства озвученного контента с использованием локальных и внешних AI-моделей.

Система предназначена для создания:

- аудиокниг;
- дубляжа;
- озвучивания документов;
- многоголосых произведений;
- голосовых библиотек;
- проектов синтеза речи.

Voxarium не является оболочкой над одной конкретной моделью синтеза речи.

AI является заменяемым компонентом архитектуры.

---

# 3. Primary Goal

Главной целью проекта является создание расширяемой платформы, а не приложения для конкретного сценария использования.

Все архитектурные решения SHALL приниматься исходя из возможности многолетнего развития проекта.

---

# 4. Long-Term Compatibility

Архитектура SHALL обеспечивать возможность развития без разрушения существующих проектов.

Изменения формата хранения данных MUST сопровождаться механизмами миграции.

Проекты предыдущих версий SHOULD открываться в новых версиях приложения.

---

# 5. Local First

Основной режим работы системы — локальный.

Архитектура MUST предполагать возможность полноценной работы без подключения к сети.

Все основные функции системы SHALL быть доступны без использования облачных сервисов.

Облачные сервисы являются дополнительными компонентами.

---

# 6. AI Independence

Архитектура НЕ должна зависеть от конкретной AI-модели.

Недопустимо проектирование системы под:

- XTTS;
- GPT-SoVITS;
- Kokoro;
- Piper;
- StyleTTS;
- любые другие конкретные реализации.

Все AI-компоненты рассматриваются исключительно как реализации контрактов.

---

# 7. Plugin First

Любой функционал, который может быть вынесен в расширение, SHOULD реализовываться как Plugin.

Core системы MUST содержать только функции, необходимые для работы платформы.

---

# 8. Deterministic Architecture

Архитектура SHALL быть детерминированной.

Одинаковые входные данные и одинаковое состояние системы MUST приводить к одинаковому результату.

Недетерминированное поведение допускается только при явном использовании внешних AI-сервисов.

---

# 9. Explicit Architecture

Все архитектурные зависимости MUST быть явными.

Недопустимо:

- скрытое создание объектов;
- неявное внедрение зависимостей;
- автоматический поиск сервисов;
- скрытая регистрация компонентов.

Каждая зависимость должна быть определена спецификацией.

---

# 10. Domain Driven Design

Предметная модель является центром архитектуры.

Application Layer зависит от Domain.

Infrastructure зависит от Application.

GUI зависит от Application.

Plugin зависят от опубликованных контрактов.

Обратные зависимости запрещены.

---

# 11. Single Source Of Truth

Каждая сущность системы имеет единственного владельца.

Каждый объект хранится только в одном месте.

Любое кэширование SHALL рассматриваться как производное представление.

---

# 12. Event Driven Collaboration

Взаимодействие компонентов SHOULD происходить посредством событий.

Прямые вызовы между независимыми подсистемами SHOULD минимизироваться.

---

# 13. Explicit State

Каждый объект системы обязан иметь явно определённое состояние.

Состояние объекта MUST быть восстанавливаемым.

Не допускается наличие скрытого состояния.

---

# 14. Immutable History

История проекта должна быть воспроизводимой.

Изменение данных SHALL фиксироваться через соответствующие механизмы проекта.

При необходимости восстановления предыдущего состояния система должна иметь возможность определить последовательность изменений.

---

# 15. Human Readable Data

Все долговременно хранимые данные SHOULD использовать открытые и документированные форматы.

Не допускается использование закрытых бинарных форматов без документированной причины.

---

# 16. Stable Public Contracts

После публикации контрактов изменение их поведения допускается только с соблюдением правил обратной совместимости.

Несовместимые изменения MUST сопровождаться новой версией контракта.

---

# 17. Replaceable Components

Любой компонент системы должен быть заменяемым без изменения архитектуры приложения.

Это относится к:

- AI;
- импортерам;
- экспортерам;
- аудиообработке;
- GUI-расширениям;
- хранилищам данных.

---

# 18. Resource Awareness

Архитектура должна учитывать ограниченность вычислительных ресурсов.

Любая операция, потенциально занимающая значительное время, SHALL выполняться асинхронно через Job Orchestrator.

---

# 19. Safety Before Performance

Корректность системы имеет приоритет над производительностью.

Недопустимо нарушение инвариантов ради увеличения скорости работы.

---

# 20. AI-Ready Specification

Настоящая спецификация разрабатывается не только для людей.

Основной целью является создание документации, по которой независимый AI-агент сможет реализовать систему без самостоятельного принятия архитектурных решений.

Любая неоднозначность SHALL устраняться путём уточнения спецификации.

---

# 21. Normative Keywords

Во всех документах используются следующие нормативные ключевые слова:

- MUST
- MUST NOT
- SHALL
- SHALL NOT
- SHOULD
- SHOULD NOT
- MAY

Их значение соответствует RFC 2119.

---

# 22. Architecture Ownership

Архитектурные решения принимаются исключительно спецификацией.

Исходный код не является источником архитектурной истины.

При противоречии между кодом и спецификацией приоритет имеет спецификация.

---

# 23. Implementation Philosophy

Разработчик или AI-агент НЕ проектирует архитектуру.

Разработчик реализует архитектуру, уже определённую настоящей спецификацией.

Если для реализации требуется принять новое архитектурное решение, значит спецификация является неполной и должна быть дополнена до начала реализации.

---

# 24. Compliance Requirements

Любой компонент Voxarium считается соответствующим архитектуре только при выполнении следующих условий:

- соблюдены все обязательные требования настоящей спецификации;
- отсутствуют запрещённые зависимости;
- соблюдены контракты;
- соблюдены инварианты предметной модели;
- соблюдены требования безопасности;
- соблюдены требования расширяемости;
- соблюдены требования обратной совместимости.

---

# 25. Compliance Checklist

Перед реализацией каждого компонента необходимо ответить положительно на следующие вопросы:

- Архитектура полностью определена?
- Все зависимости документированы?
- Все состояния описаны?
- Все события определены?
- Все публичные интерфейсы определены?
- Все инварианты перечислены?
- Все ограничения реализации описаны?
- Не требуется самостоятельное архитектурное проектирование?

Если хотя бы на один вопрос получен отрицательный ответ, соответствующая часть спецификации должна быть дополнена.
