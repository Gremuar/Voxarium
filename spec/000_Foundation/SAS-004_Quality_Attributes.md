# 1. Purpose

Настоящий документ определяет нефункциональные требования (Quality Attributes) платформы Voxarium.

Все архитектурные решения должны оцениваться относительно настоящего документа.

При наличии нескольких допустимых технических решений предпочтение должно отдаваться решению, которое обеспечивает лучшее соответствие данным требованиям.

---

# 2. Priority

Атрибуты качества имеют следующий приоритет.

| Priority | Attribute |
|-----------|-----------|
| P1 | Correctness |
| P1 | Reliability |
| P1 | Maintainability |
| P2 | Extensibility |
| P2 | Testability |
| P2 | Recoverability |
| P3 | Performance |
| P3 | Usability |
| P3 | Portability |
| P4 | Resource Efficiency |

Оптимизация менее приоритетного атрибута не должна ухудшать более приоритетный.

---

# 3. Correctness

Correctness является главным качественным атрибутом платформы.

Система должна выполнять только корректные операции.

При невозможности безопасного выполнения операция должна завершаться ошибкой.

Частично корректный результат считается некорректным.

---

# 4. Reliability

Платформа должна сохранять работоспособность при возникновении ошибок отдельных компонентов.

Отказ одного Plugin не должен приводить к отказу Core.

Отказ одного Workflow не должен приводить к остановке остальных Workflow.

Повреждение временного кэша не должно приводить к потере пользовательских данных.

---

# 5. Recoverability

После аварийного завершения приложения пользователь должен иметь возможность продолжить работу.

Минимальные требования:

- автоматическое обнаружение незавершённых операций;
- автоматическое восстановление проекта;
- сохранение истории ошибок;
- восстановление состояния Workflow;
- восстановление очередей генерации.

---

# 6. Maintainability

Архитектура должна обеспечивать возможность сопровождения проекта в течение длительного времени.

Добавление новой функциональности не должно требовать переработки существующих модулей.

Все публичные контракты должны быть документированы.

Каждый модуль должен обладать единственной зоной ответственности.

---

# 7. Extensibility

Расширение платформы должно осуществляться преимущественно посредством Plugin.

Добавление нового Speech Engine, Importer, Exporter или Voice Cloner не должно требовать изменения Core.

Архитектура должна допускать появление новых категорий Plugin без изменения существующих модулей.

---

# 8. Testability

Все компоненты платформы должны быть пригодны для автоматического тестирования.

Каждый публичный контракт должен иметь тестовую реализацию.

Каждый модуль должен тестироваться независимо.

Архитектурные проверки должны быть автоматизированы.

---

# 9. Performance

Производительность является важным, но не доминирующим качественным атрибутом.

Не допускается повышение производительности ценой нарушения архитектурных принципов.

Оптимизации должны быть локальными и обратимыми.

---

# 10. Responsiveness

Графический интерфейс должен оставаться отзывчивым.

Длительные операции запрещается выполнять в UI-потоке.

Индикация выполнения должна обновляться в режиме реального времени.

Пользователь должен иметь возможность отменить поддерживаемые длительные операции.

---

# 11. Scalability

Архитектура должна поддерживать рост:

- количества проектов;
- размера проектов;
- числа документов;
- числа Fragment;
- числа Voice Profile;
- количества Speech Segment;
- числа установленных Plugin.

Рост объёма данных не должен требовать изменения архитектуры.

---

# 12. Resource Efficiency

Платформа должна эффективно использовать вычислительные ресурсы.

Не допускается бесконтрольное потребление:

- оперативной памяти;
- видеопамяти;
- дискового пространства;
- процессорного времени.

Освобождение временных ресурсов должно происходить автоматически после завершения операций.

---

# 13. GPU Utilization

При наличии совместимого оборудования платформа должна предоставлять возможность использования GPU.

Отсутствие GPU не должно препятствовать работе платформы.

Выбор вычислительного устройства должен определяться используемым Plugin.

Core не должен содержать логики управления конкретными GPU API.

---

# 14. Local-First

Локальное выполнение является основным режимом работы платформы.

Все основные функции должны быть доступны без подключения к сети.

Использование облачных сервисов является дополнительной возможностью и реализуется через Plugin.

Core не должен зависеть от удалённых сервисов.

---

# 15. Portability

Платформа должна поддерживать перенос проектов между поддерживаемыми операционными системами без преобразования данных.

Поддерживаемые платформы:

- Windows
- Linux

Архитектура должна учитывать возможность добавления поддержки macOS без изменения предметной модели.

---

# 16. Compatibility

Проекты, созданные предыдущими совместимыми версиями платформы, должны открываться без потери пользовательских данных.

При невозможности автоматической миграции пользователь должен получить понятное диагностическое сообщение.

---

# 17. Security

Core не должен выполнять произвольный код Plugin без прохождения процедуры регистрации и проверки совместимости.

Каждый Plugin должен явно декларировать требуемые разрешения.

Все операции записи должны выполняться только в пределах разрешённых каталогов проекта или пользовательских каталогов, выбранных пользователем.

---

# 18. Observability

Все критические операции должны быть доступны для анализа.

Минимально обязательны:

- журнал ошибок;
- журнал Workflow;
- журнал загрузки Plugin;
- журнал импорта;
- журнал генерации;
- журнал экспорта.

Средства диагностики не должны изменять состояние предметной модели.

---

# 19. Determinism

Поведение Core должно быть детерминированным.

Недетерминированность допускается только в компонентах, использующих модели искусственного интеллекта.

Такое поведение должно быть явно задокументировано соответствующим Plugin.

---

# 20. Data Integrity

Все пользовательские данные должны сохранять внутреннюю согласованность.

Любая операция записи должна завершаться одним из двух состояний:

- операция успешно завершена;
- операция полностью отменена.

Частичное изменение состояния проекта запрещается.

---

# 21. Availability

Платформа должна сохранять возможность выполнения независимых операций при отказе отдельных компонентов.

Недоступность одного Speech Engine не должна препятствовать использованию остальных Speech Engine.

Недоступность одного Plugin не должна препятствовать работе остальных Plugin.

---

# 22. Upgradeability

Обновление платформы не должно требовать пересоздания существующих проектов.

Миграция формата проекта должна выполняться отдельным механизмом.

Все изменения формата проекта должны быть обратимо документированы.

---

# 23. Documentation Quality

Каждый публичный архитектурный элемент должен иметь нормативную документацию.

Документация является частью архитектуры.

Изменение архитектуры без соответствующего изменения документации запрещается.

---

# 24. Measurability

Каждый качественный атрибут должен иметь объективный способ проверки.

Если атрибут невозможно проверить автоматически или документально, он не считается выполненным.

Документы раздела `900_Testing` определяют методики проверки каждого требования настоящего документа.

---

# 25. Compliance

Все архитектурные решения, модули, Plugin, тесты и пользовательские интерфейсы обязаны соответствовать требованиям настоящего документа.

Любое исключение должно быть оформлено отдельным Architecture Decision Record.
