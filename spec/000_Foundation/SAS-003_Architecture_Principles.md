# 1. Purpose

Настоящий документ определяет обязательные архитектурные принципы платформы Voxarium.

Настоящий документ определяет допустимые зависимости между архитектурными слоями, правила построения модулей, взаимодействия компонентов и ограничения, обязательные для всей системы.

Любое отклонение от требований настоящего документа считается архитектурным нарушением.

---

# 2. Architectural Style

Архитектура платформы строится на сочетании следующих подходов.

- Hexagonal Architecture
- Modular Monolith
- Domain-Driven Design
- CQRS
- Event-Driven Architecture
- Dependency Injection
- Plugin-Oriented Architecture

Все перечисленные подходы являются обязательными.

Замена любого из них допускается исключительно через ADR.

---

# 3. Architectural Goals

Архитектура должна обеспечивать:

- независимость предметной модели;
- долгосрочную сопровождаемость;
- простоту тестирования;
- расширяемость без изменения Core;
- возможность автоматической генерации кода;
- возможность подключения новых AI-моделей;
- переносимость между операционными системами;
- локальное выполнение без подключения к сети.

---

# 4. Architectural Layers

Архитектура разделяется на следующие уровни.

```
Presentation

↓

Application

↓

Domain

↓

Ports

↓

Infrastructure
```

Направление зависимостей допускается только сверху вниз.

---

# 5. Layer Responsibilities

## Presentation

Отвечает исключительно за взаимодействие с пользователем.

Не содержит бизнес-логики.

Не обращается напрямую к Infrastructure.

---

## Application

Координирует выполнение пользовательских сценариев.

Не содержит правил предметной области.

Не зависит от GUI.

---

## Domain

Содержит предметную модель.

Не зависит ни от одного другого слоя.

Является центром архитектуры.

---

## Ports

Определяет публичные контракты взаимодействия.

Не содержит реализации.

---

## Infrastructure

Содержит технические реализации.

Является заменяемым слоем.

---

# 6. Dependency Rule

Все зависимости обязаны быть направлены к Domain.

Domain не может зависеть ни от одного другого слоя.

Infrastructure зависит от Ports.

Application зависит от Domain и Ports.

Presentation зависит от Application.

---

# 7. Forbidden Dependencies

Запрещаются следующие зависимости.

Presentation → Infrastructure

Presentation → Database

Presentation → Plugin

Domain → GUI

Domain → SQLite

Domain → Filesystem

Domain → Network

Domain → Plugin

Domain → AI Engine

Application → Qt

Application → PySide

Plugin → Internal Core Classes

Infrastructure → GUI

Infrastructure → Presentation

Любая другая зависимость, нарушающая правило направленности, также запрещается.

---

# 8. Module Independence

Каждый модуль Core обязан быть независимым.

Модуль не имеет права использовать внутренние классы другого модуля.

Взаимодействие модулей осуществляется исключительно через опубликованные контракты.

---

# 9. Module Responsibilities

Каждый модуль обязан иметь одну основную ответственность.

Модуль не должен объединять несколько независимых предметных областей.

Если модуль требует более одной причины для изменения, он должен быть разделён.

---

# 10. Ports

Каждый внешний механизм подключается через Port.

Примеры:

Speech Engine

Storage

Importer

Exporter

Voice Cloner

Translation

Playback

Audio Processing

Ports принадлежат Core.

Реализации Ports принадлежат Infrastructure или Plugin.

---

# 11. Adapters

Adapter реализует Port.

Adapter преобразует внешний интерфейс во внутренний контракт.

Adapter не содержит бизнес-логики.

---

# 12. Dependency Injection

Создание зависимостей вручную запрещается.

Все зависимости предоставляются контейнером внедрения зависимостей.

Использование глобальных Singleton запрещается, кроме специально документированных инфраструктурных компонентов.

---

# 13. CQRS

Все операции разделяются на:

Command

Query

Command изменяет состояние.

Query возвращает данные.

Query не должна изменять состояние системы.

---

# 14. Event-Driven Architecture

Все изменения состояния сопровождаются публикацией Domain Event.

События являются неизменяемыми.

Подписчики не должны изменять опубликованное событие.

События могут иметь произвольное количество подписчиков.

Отправитель события не знает своих подписчиков.

---

# 15. Message Bus

Все Commands, Events и Queries передаются через Message Bus.

Прямой вызов между независимыми модулями запрещён, если для него существует соответствующий контракт Message Bus.

Message Bus является инфраструктурным сервисом Core.

---

# 16. Plugin Architecture

Plugin является внешним компонентом системы.

Plugin не входит в состав Core.

Plugin взаимодействует с платформой только через Ports.

Plugin не имеет доступа к внутренним структурам Core.

Plugin не может изменять предметную модель напрямую.

---

# 17. Runtime

Runtime отвечает за:

- загрузку модулей;
- загрузку Plugin;
- управление жизненным циклом;
- создание контейнера зависимостей;
- запуск Workflow;
- завершение работы системы.

Runtime не содержит бизнес-логики.

---

# 18. State Management

Единственным источником истины является предметная модель.

GUI не хранит собственное состояние предметной области.

Plugin не хранит состояние предметной области.

Кэш не является источником истины.

---

# 19. Transactions

Любое изменение предметной модели должно выполняться как атомарная операция.

При невозможности завершить операцию система обязана откатить изменения до согласованного состояния.

Требования к реализации механизма транзакций определяются документами раздела `600_Project_Format`.

---

# 20. Error Isolation

Ошибка одного модуля не должна приводить к остановке остальных модулей.

Ошибка Plugin не должна завершать работу Core.

Ошибка одного Workflow не должна останавливать остальные Workflow.

---

# 21. Threading Model

Предметная модель должна считаться потоконебезопасной.

Одновременное изменение одной и той же сущности несколькими потоками запрещается.

Средства синхронизации являются обязанностью Application Layer.

Domain Layer не должен содержать примитивов синхронизации.

---

# 22. Background Operations

Все длительные операции должны выполняться вне пользовательского интерфейса.

К длительным операциям относятся:

- импорт;
- экспорт;
- генерация речи;
- клонирование голоса;
- анализ текста;
- индексирование;
- построение кэша;
- миграция проекта.

---

# 23. Determinism

Одинаковые входные данные должны приводить к одинаковому поведению системы.

Если внешняя модель искусственного интеллекта допускает недетерминированность, этот факт должен быть явно отражён в соответствующем Plugin.

---

# 24. Extensibility

Каждый новый механизм платформы должен быть реализуем без изменения существующих модулей Core.

Если добавление функциональности требует изменения нескольких существующих модулей, архитектурное решение считается неудовлетворительным и подлежит пересмотру.

---

# 25. Observability

Все значимые архитектурные операции должны быть наблюдаемыми.

Минимально наблюдаемыми являются:

- запуск приложения;
- открытие проекта;
- импорт;
- генерация;
- экспорт;
- ошибки Plugin;
- ошибки Workflow;
- загрузка модулей;
- загрузка Plugin.

Механизм наблюдаемости определяется модулем Logging.

---

# 26. Testability

Каждый модуль должен быть пригоден для изолированного тестирования.

Любой внешний компонент должен заменяться тестовой реализацией через соответствующий Port.

Использование реальных AI-моделей, файловой системы или сетевых сервисов в модульных тестах запрещается.

---

# 27. Architectural Compliance

Архитектура проекта должна автоматически проверяться.

Минимальный перечень проверок:

- отсутствие запрещённых зависимостей;
- отсутствие циклических зависимостей между модулями;
- отсутствие обращений к внутренним API других модулей;
- соответствие структуре каталогов;
- соответствие правилам именования;
- соответствие публичным контрактам.

Проверки должны быть частью конвейера непрерывной интеграции.

---

# 28. Compliance

Все архитектурные решения платформы обязаны соответствовать требованиям настоящего документа.

Любое исключение оформляется отдельным ADR с указанием причин, анализа альтернатив, последствий и срока действия исключения.
